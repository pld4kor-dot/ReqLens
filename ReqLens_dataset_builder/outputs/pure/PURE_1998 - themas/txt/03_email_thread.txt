Two QA points:
1) Please make the “only SHALL is binding” statement explicit early. We’ve had audits where ambiguous language causes grief.
2) Temperature inputs (both the live temp and the setpoint value) must be validated against the allowed range. If invalid, mark as such and do not use for control decisions. This affects both runtime logic AND the logs that feed the history report.

Also, control outputs: THEMAS generates H/C ON/OFF requests. We do not and cannot determine whether the unit actually did it. The software keeps the intended ON/OFF status per unit to drive utilization, but no execution confirmation is expected. This needs to be spelled out so no one tries to write negative tests expecting acknowledgments.

On utilization: ensure the SRS calls out the cap on concurrent units and the LIFO queue for excess requests, and what happens when a unit goes OFF — we pull from the queue, re-check the thermostat need, and only then send an ON. If we can’t honor an ON because we’re at the cap, log a “Unit Unavailable” event.

Reports specifics look good:
- 12 mo history pulls all events sorted by date/time.
- monthly stats includes per-day % ON time per unit and, per thermostat, grant vs deny ratios.
- operator selects month from an available list.
- operator picks path/name; files are ASCII.

Final admin: SRS includes refs. Conflicts: SOW > SRS > refs. Tech standards are generally available (libraries). Non-public copies via Barry. IEEE 830 compliance + Section 2/3 content to be stated. SRS is the basis for design. Scope: software requirements only, hardware lives elsewhere.

All, adding a couple of clarifications (sorry long email):

- External Interfaces: We’ll identify (1) thermostat input interface (two data items: current temp and desired setpoint), (2) unit command output interface (emit H/C ON/OFF requests), and (3) supervisor console UI. We will not define exact external system vendors in the SRS. Communication with devices happens only via those interfaces.

- Platform: per Jen, Windows NT across the board — runtime, IPC, and device interfaces. I’ll tighten the language to avoid implying portability.

- Simulator: We will include a requirement to produce a thermostat simulator prototype for V&V of control logic reliability. That way QA can test decisions without field hardware.

- Console functions (per Nina): list thermostats with current settings; allow selecting one and choosing a valid setpoint (from a constrained list); manual ON/OFF controls.

- Temperature validation: both the thermometer reading and the setpoint are checked against the valid range. Valid values are applied; invalid are flagged and excluded from control decisions.

- Control decisions: we’ll point to the rules sections (3.1.4.1/3.1.4.2). We can mention the trigger/limit concepts, but the details live in those rule sections to avoid duplication.

- Utilization & Queue: we keep a software ON/OFF state, cap concurrent units, use LIFO for waiting requests, and on OFF we try to service the latest queued item after re-checking the thermostat’s need. If we are at capacity when a new ON is requested, we emit a Unit Unavailable event.

- Reports:
   • Operational history report: include all events from the event DB, sort by date/time (oldest-to-newest), produce ASCII file.
   • Statistical monthly report: present available months for selection; upon selection, produce ASCII file including per-day % ON time per unit and, by thermostat, grant/deny ratios (grant = request honored; deny = request not honored due to cap). Operator chooses path and filename when saving both reports.

- Language & Structure: Only clauses using “shall” are binding requirements. Document conforms to ANSI/IEEE 830 with the expected sections: Section 2 (product perspective, functions, user characteristics, constraints, assumptions/dependencies) and Section 3 (functional/performance requirements, design constraints, attributes, external interface requirements) needed by the Design Requirements team and architecture.

- Scope & Precedence: SRS is limited to software requirements. It will be used as the basis for the software design. Hardware requirements remain in the system-level spec. Include references as part of the spec; where conflicts exist, SRS text takes precedence over references, except the SoW (THEMAS_SOW_001) supersedes the SRS. Standards are assumed accessible via libraries; other referenced materials available via Barry.

Priya
