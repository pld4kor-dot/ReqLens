Hi all,

Quick sanity check:
- We’re planning to prompt users once at the start for credentials (Gatekeeper), then NOT ask again when they run Daymet or Biome-BGC jobs or when downloading. Delegated credentials should cover app->compute->storage pathways.

- We’ll surface downloads on dataset-bearing objects (e.g., Grid Registration, Daymet outputs).

Thanks everyone for the quick replies.

Acknowledged and incorporated:
- Gatekeeper with single credential prompt at session start; no extra prompts later. We will use delegated credentials.
- Use Globus toolkit for secure transfers and remote job integration. All communications to compute/storage over secure channels.
- Compute on Hemisphere nodes only; portal host stays lightweight. Nate is helping with endpoints.
- All file-based storage -> NCAR Mass Storage. Scratch is only for staging on the compute side.
- Daymet: workflow to convert surface observations into gridded weather via projects. Outputs downloadable and feed Biome-BGC inputs.
- Disturbance modeling: support disturbance projects and events; events (fires + deforestation) are single timestamps; unlimited events per project.
- Domain objects CRUD: Projection, Grid Registration, Surface Observation, Site Data, Daymet Parameterization, DEM, Analysis Mask, Daymet Project/Output, PFT, BiomeBGC Site Data, Output Specification. We’ll keep clear relationships so runs show which inputs they used.
- Sharing/merging objects with provenance; collaboration across projects.
- Templates: create/view/edit/submit/delete; access control; intended for reuse.
- Visualization: very basic previews for Surface Obs, DEM, Analysis Mask, Site Data — low priority.
- Roles: Scientists (favored) can run models; Data Users read-only + downloads; Admins manage accounts/roles and can unlock when needed.

Open items (still):
- Minor: presentation of Daymet output previews vs full download.

If I missed anything, shout. Otherwise, we’ll move this into the spec draft and coordinate with the NCAR DataPortal team on integration points.
