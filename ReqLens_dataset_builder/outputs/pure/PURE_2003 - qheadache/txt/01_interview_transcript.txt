Sasha [Community Rep]: And they will use both. Some will have trackpads, some mice, even those weird pen devices. But basically move the cursor and click. Oh, and typing names when they win.

Lars [Qt Programmer]: Input devices that can move a cursor and accept text entry should work fine in Qt: mouse, touchpad, trackball, pen, even some joystick-driven cursors technically. Keyboard for menu navigation and name entry is trivial.

Lars [Qt Programmer]: Got it. 5x tall, 4x wide. And the unit x is our layout scaling unit — constrained so that x stays between 50 and 100 pixels. We’ll pick an initial x based on the window size.

Victor [Product Owner]: We’re also saying it’s playable at 800x600 minimum. That’s non-negotiable. It needs to be clear at that resolution.

Renee [QA Tester]: I’m going to test on a Windows machine at 800x600 just to be sure, and then on a couple of Linux variants. But we have to run on Windows; folks keep asking for Windows builds in the community.

Lars [Qt Programmer]: Running on all Qt-supported OSes is fine. Windows, Linux, macOS — but Windows portability is an explicit must-have.

Sasha [Community Rep]: Another thing people care about — visuals. We promised a black backdrop for the board, right? And the blocks are the bright yellow ones.

Victor [Product Owner]: Yes, black board, yellow blocks. Keep the blocks with a tiny spacing so they don’t stick to each other visually.

Maya [Business Analyst]: Spacing is 0.1x between blocks. It’s subtle but noticeable.

Lars [Qt Programmer]: Okay, 0.1x gutters. I’ll enforce that consistently. Now, the pieces: remind me of the exact set.

Maya [Business Analyst]: Four small squares, each 1x by 1x.

Renee [QA Tester]: And four tall rectangles, 2x tall by x wide. Plus one wide rectangle, x tall by 2x wide, and the “big square,” which is 2x by 2x.

Sasha [Community Rep]: That’s the classic set folks expect. Don’t mess with the counts or the sizes — there was drama in another game when they changed it.

Victor [Product Owner]: We’re not changing it. It’s fixed.

Lars [Qt Programmer]: Alright. Play interaction: we’re going with mouse-driven drag, but I want to confirm: when the player presses the left mouse button on a block, does that immediately put us into a moving state?

Maya [Business Analyst]: Yes. We’ve been calling it “block movement” state in notes. You’re dragging at that point.

Renee [QA Tester]: And when they let go of the left button over that block, we’re back to selection mode. That transition matters for testing stuck states.

Sasha [Community Rep]: People will expect the block to stick to the cursor while they’re holding it down. If it lags or jumps, they’ll complain.

Lars [Qt Programmer]: It’ll follow the mouse movements smoothly; I’ll make sure the drag is responsive.

Victor [Product Owner]: Also, we have to do menus. The main window has a menu bar — Stats, Help, the usual. Keep it straightforward, menu-driven. No hidden gestures or anything.

Maya [Business Analyst]: Right, graphics and menus. We can have some sound effects if you want, but the game must be fully playable muted. Actually, let’s make sure that if audio is off, nothing breaks.

Renee [QA Tester]: I’ll test with the OS muted too. I don’t want sound to be a dependency. We had that issue somewhere else where a missing audio driver froze a dialog. Not again.

Sasha [Community Rep]: And kids play on school computers with sound off; also some parents hate bleeps and bloops.

Lars [Qt Programmer]: Menus: Stats should have two items — Display and Erase — in that order? I’ve seen both orders in screenshots from other apps.

Maya [Business Analyst]: Display, then Erase. When you choose Display, it pops up the scores window listing recorded players and their times and moves. If you choose Erase, it clears all stats.

Renee [QA Tester]: And the menu should close after the action is done. I don’t want sticky open menus confusing users.

Victor [Product Owner]: Yes, close the menu afterward. Same idea for Help -> About. Open About, then when you’re done, the Help menu shouldn’t stay open.

Lars [Qt Programmer]: About content — do we have the actual text? I need to embed it.

Maya [Business Analyst]: We do. The exact line is: “Qheadache 1.0 by Jean-Philippe Brossat jp_brossat@yahoo.fr”. That needs to be shown exactly like that in the About window.

Sasha [Community Rep]: It’s a nice touch to credit the original author. People in the community care about attribution.

Renee [QA Tester]: Okay, I’ll verify the exact characters, spaces, the email, everything. Now, stats: on victory we prompt for the player’s name, right? That’s when we record the time and action count together with the name.

Victor [Product Owner]: Yes. So you solve the puzzle, you get a name prompt. If it’s a kid, they might just type initials. But we need a field for it.

Lars [Qt Programmer]: We’ll have a modal dialog on victory with an entry field. Then we write out the score record: name, elapsed time, and action count.

Sasha [Community Rep]: People will want to see a list. So the scores window you mentioned needs to show all saved players sorted… actually, do we care about ordering?

Maya [Business Analyst]: Let’s not overcomplicate it right now. Just show the list. Sorting isn’t a requirement this round. The important part is we display all recorded scores.

Renee [QA Tester]: Undo/redo: we agreed to allow undoing and redoing a lot of moves. What’s the cap again?

Lars [Qt Programmer]: I can implement a ring buffer. But what’s the number we picked?

Maya [Business Analyst]: Up to the last 1000 actions. It’s high, but it stops memory growth.

Sasha [Community Rep]: 1000 is fine. People will brag they can step through their entire solution backward.

Victor [Product Owner]: That number’s a ceiling. We aren’t exposing that as a setting — at least not now.

Renee [QA Tester]: So action count is the number we present on screen during play too, right? I want to see it increment so kids can see how efficient they were.

Lars [Qt Programmer]: Yes, I’ll surface both the elapsed time and the current move count somewhere in the main window UI — maybe a small status bar or a panel near the menu.

Maya [Business Analyst]: Make sure it’s readable at 800x600. Big clear digits. We’re designing for clarity, no clutter.

Sasha [Community Rep]: UI question: The board area itself is a black rectangle of 5x by 4x. If the window resizes, do we scale x between 50 and 100? What if the window gets huge?

Lars [Qt Programmer]: I’ll clamp x within 50–100 pixels. If the window is too big, we can keep the board centered with padding. If it’s too small to fit 50px per x, we block it — but the minimum app window will be at least 800x600 anyway, so it should be fine.

Renee [QA Tester]: I’ll test resizing scenarios. Also DPI scaling on Windows. But at base level, as long as our math on x is consistent, the blocks won’t collide visually thanks to that 0.1x gap.

Victor [Product Owner]: And the highlight or movement visuals: we’re not doing fancy shadows or filters. Keep it crisp. Yellow on black with clean spacing is the look.

Sasha [Community Rep]: Does the block actually “snap” to anything? People will complain if the drag sticks inside other blocks.

Lars [Qt Programmer]: It won’t overlap others. Movement is constrained so blocks slide freely where allowed; while moving, the selected block follows the cursor. When you release the left button, we switch back to selection. That’s the cycle.

Maya [Business Analyst]: Exactly. Press-down on a block puts you in moving mode; release returns you to normal. That matches what players expect.

Renee [QA Tester]: Let’s not forget menu operations are keyboard accessible too. Arrow keys and Enter should work to select Stats->Display, Stats->Erase, Help->About.

Lars [Qt Programmer]: Standard Qt menu input covers that. Cursor input from keyboard and pointing devices is a must anyway.

Victor [Product Owner]: One more time on brand: the name Qheadache needs to appear in the title bar of the main window. Also in the About line we discussed.

Maya [Business Analyst]: And in the installer name and icon if we do one. But at minimum inside the app.

Victor [Product Owner]: Menu closure: after you pick Display, the menu should disappear and leave you with the scores window. Don’t leave the menu stuck open.

Maya [Business Analyst]: Let’s wrap with a quick recap: Qheadache is a desktop Qt game with a main window that has a menu bar and the puzzle board. The board is 5x tall, 4x wide, black background; blocks are yellow, spaced 0.1x apart. There are four 1x1 squares, four 2x by 1x rectangles, one 1x by 2x rectangle, and one 2x by 2x square. Play is mouse-driven: press left to enter movement, the block follows the cursor, release returns to selection. Keyboard and pointing devices work for cursor and menu and name entry. We track time and actions, show those during play, and on victory we prompt for a name and store that with the time and move count. Stats menu shows Display then Erase; Display opens the scores window listing all players; Erase clears stats; menus close after the action. Help->About opens an About window with the exact text — and the menu closes too. The app runs standalone across Qt-supported OSes including Windows. The UI is clear at 800x600. No audio dependency. Single local user. Audience is 8+ up to adult enthusiasts.
