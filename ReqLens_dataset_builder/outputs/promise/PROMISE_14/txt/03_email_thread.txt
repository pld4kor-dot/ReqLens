Jumping in with ops constraints so we’re not overpromising:

- Performance: We’re targeting that every user transaction (think add, drop, trade confirmations, etc.) completes over the network in about a quarter of a second. 250 ms on the server side is the SLO we’re drafting. That means we’ll need to keep payloads tight — e.g., limiting the league wall to five posts helps a lot.

- Availability: The business has floated a minutes-per-year downtime goal. I’ve penciled in a 10 minutes/year upper bound outside the scheduled Wednesday window. It’s aggressive but we’ll plan around it.

- Hosting: This is a web application server deployment. Just clarifying we’re not doing any native app in scope right now.

- Maintenance: Agree on the 3–6 AM CST Wednesday in-season updates. Season-start major version is fine. If we need an emergency outside that slot, we will coordinate and avoid Sundays.

Alex — keeping actions and nav separated and a breadcrumb on each page both reduce errors, so I’m on board.

Adding DB angle:

- We will keep a primary database focused on active users’ needs. It will act like a cached store for hot data — current rosters, pending add/drop/trade transactions, current week stat lines. That’s how we can realistically hit the ~250 ms target without recomputing.

- A secondary database will hold the long-term historicals: prior seasons’ stats, career numbers, old message wall posts, long-term league records. We’ll archive into this regularly so the primary stays lean.

- For the message wall, I’ll index by league and timestamp in the primary to fetch exactly the last five quickly. Teaser length we can cap around 150 chars at the app layer.

- Points will be included in the stat line object we serve to the UI so that wherever players show up, they get 25+ stats plus points without extra calls.
