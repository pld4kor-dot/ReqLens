TYPE  : interview_transcript
TITLE : Kickoff Interview on EVLA Correlator Control System Requirements
UNIT  : PURE_2002 - evla corr
============================================================

Alice [Project Scientist]: So, to start off, we really need the system to handle all the correlator configuration and real-time monitoring and control. It's crucial that it also supports hardware testing and servicing. Bob [Software Developer]: Right, so we're talking about a system that not only configures the correlator but also provides interfaces for hardware diagnostics? Alice [Project Scientist]: Exactly. And it should fit seamlessly into the overall EVLA Monitor and Control infrastructure. Carol [System Administrator]: When you say 'fit seamlessly,' do you mean it has to follow existing network protocols and standards? Alice [Project Scientist]: Yes, it must integrate fully with the EVLA M&C environment. Also, we want it to modularize the correlator system, so it's easier to manage within the EVLA setup. Bob [Software Developer]: That suggests we need some abstraction layer to separate correlator-specific functions from the rest of the system. Alice [Project Scientist]: Precisely. And for the interface, we want a Virtual Correlator Interface, or VCI, acting as a software gateway on the Master Correlator Control Computer, or MCCC. Dave [Hardware Engineer]: So the MCCC will translate those commands into hardware setup instructions? Alice [Project Scientist]: Yes, the VCI should translate configuration commands from the EVLA M&C system into specific hardware configuration tables. Carol [System Administrator]: What about the network architecture? Any preferences? Dave [Hardware Engineer]: We plan to implement a master-slave network. One master computer will coordinate multiple slave processors that handle the correlator hardware directly. Bob [Software Developer]: So the master handles most of the monitor/control interface with external systems, and slaves manage real-time computing for hardware control? Dave [Hardware Engineer]: Exactly. This separation will help with serviceability and programmability because the correlator hardware will be isolated from the EVLA M&C environment. Carol [System Administrator]: Isolation is good for security and maintenance. Speaking of which, do we need redundancy? Alice [Project Scientist]: Absolutely. Critical areas must be redundant to ensure high availability. Also, the system should be highly modular to facilitate maintenance. Bob [Software Developer]: For monitoring, the correlator monitor subsystem should provide system-wide access to all correlator states, including the M&C supervisor state. Alice [Project Scientist]: Yes, and some information must be available on a time-synchronous basis for other systems like the backend and archives, while other data can be provided on request. Carol [System Administrator]: Access to information will be limited by hardware, bandwidth, and security policies, right? Alice [Project Scientist]: Correct. Also, error and status messages need to be concise, time- and location-referenced, and controllable in content when reported to upper system levels. Dave [Hardware Engineer]: Regarding hardware control, the system should accept correlator configurations from EVLA M&C, translate them into goal-oriented hardware tables, and support a GUI for human operators using the same data structures. Bob [Software Developer]: That means the VCI will have both software and GUI interfaces for configuration. Alice [Project Scientist]: Exactly. And the system must provide specific data sets to the Backend Data Processing System over a secondary virtual network, ensuring timely and robust delivery. Carol [System Administrator]: What happens if the network connection to EVLA M&C is lost temporarily? Dave [Hardware Engineer]: The system will spool ancillary monitor data, including health and error messages, to prevent data loss during outages. Bob [Software Developer]: Also, data sample rates and contents should be fully controllable either via EVLA M&C or the backend processing controller. Alice [Project Scientist]: Right. And the system should automatically attempt recovery from hardware failures or hot-swapped devices. Dave [Hardware Engineer]: If a Correlator Monitor Interface Board (CMIB) fails and doesn't respond to reboot or self-healing, the system must alert for hardware repair and try to restart and reconfigure it automatically. Carol [System Administrator]: For system health, the MCCC will be monitored by internal watchdogs and external systems like the Correlator Power Control Computer (CPCC). Bob [Software Developer]: And if the MCCC fails beyond recovery, the backup MCCC should be activated automatically via CPCC or by human intervention. Alice [Project Scientist]: Both primary and secondary MCCC systems must maintain full CMCS state information so that communication can reroute to the backup if needed. Carol [System Administrator]: What about maintenance? We need tools to inspect and monitor individual CMIB devices remotely and trace faults to specific hot-swappable subsystems. Dave [Hardware Engineer]: Also, twisted pair copper cabling will be used unless noise or physical layout requires fiber optic or other low-RFI materials, especially for shielded room penetrations. Bob [Software Developer]: To reduce wiring complexity, network switches will be employed within correlator racks. Carol [System Administrator]: And at the MCCC interface level, routers and switches will help protect against unauthorized access and irrelevant traffic. Alice [Project Scientist]: The CMIBs should have unique IP addresses formed from a 16-bit identifier read from the host board, allowing IP addressing to carry over for hot-swap modules. Dave [Hardware Engineer]: For physical status, there should be visible indicators on the CMIB showing green for normal operation and red for faults. Bob [Software Developer]: The hardware platform should have at least 64 MB SDRAM, IDE hard disk interface, serial and parallel ports, support PCI and ISA buses, and a 100BaseT network interface. Carol [System Administrator]: The system will boot and run a commercial off-the-shelf OS in a near real-time environment from local storage, supporting real-time control, diagnostics, standalone test bench operation, and remote upgrades of hardware PLD/FPGA personalities. Alice [Project Scientist]: Multiple Ethernet interfaces and server/host services will be supported. The system should be a high-availability general-purpose computer, possibly hot-swappable or redundant, with self-healing capabilities where possible. Dave [Hardware Engineer]: It must accept many external hardware status signals directly or through interfaces. Carol [System Administrator]: The system should have local disk and file system facilities to boot and run standalone, allowing power monitoring and control during M&C network failures. Bob [Software Developer]: All messages between CMCS layers will include UTC and wall clock timestamps, with error messages stamped at discovery time and control messages at generation time. Alice [Project Scientist]: Authorized users should have full access to all messaging, monitoring, and control traffic for testing, debugging, or offline control. Carol [System Administrator]: The system will detect processor hardware failures, OS hangs, temperature or voltage deviations, performance below specs, error rates above limits, and internal communication failures. Dave [Hardware Engineer]: It will also detect external communication disruptions with EVLA M&C and report abnormal conditions. Bob [Software Developer]: The system should automatically take corrective actions to mitigate detected issues. Alice [Project Scientist]: Security is critical. Unauthorized users must be prevented from accessing the system. Only software and hardware development, testing, maintenance, and operations personnel should be authorized. Carol [System Administrator]: Users will be uniquely identified by username and password, with authentication and authorization before access is granted. Bob [Software Developer]: Access attempts will be logged for audit purposes. Systems operations personnel will have unrestricted access and can grant or revoke privileges per user. Alice [Project Scientist]: Development, testing, and maintenance staff will have restricted access appropriate to their job needs. Carol [System Administrator]: Specifications and configuration information must be readily available. Dave [Hardware Engineer]: The system will receive configuration info from EVLA M&C and translate it into physical hardware setups, processing and transferring dynamic control and monitor data. Bob [Software Developer]: It will monitor correlator health and autonomously recover from faults where possible. Alice [Project Scientist]: Limited real-time data processing and probing will be supported, including tools to collect and display auto-correlation products. Carol [System Administrator]: The system should allow easy access to aid testing and debugging without affecting hardware or software interfaces. Dave [Hardware Engineer]: That covers the main points. Any questions? Bob [Software Developer]: Just to confirm, the master computer handles most external monitor/control interfaces, and slaves handle real-time computing for hardware under their control? Alice [Project Scientist]: Yes, that's correct. Carol [System Administrator]: And isolation of correlator hardware from EVLA M&C environment is to ease serviceability and programmability? Alice [Project Scientist]: Exactly. Thanks everyone for the input. We'll document these points for the requirements specification.
