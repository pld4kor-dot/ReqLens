Discussion — Objects and Projects
- Projects vs objects: objects = reusable data elements; projects = grouping for runs/controls; keep three object categories.
- CRUD coverage required for: Projection, Grid Registration, Surface Observation, Site Data, Daymet Parameterization, DEM, Analysis Mask, Daymet Project, Daymet Output, Plant Functional Type (PFT), BiomeBGC Site Data, Output Specification.
- Rob: Implement create/read/update/delete for all above. Where delete is unsafe, restrict it; we’ll document the rules.
- Sharing & collaboration: objects can be shared across projects; merge workflow for compatible types (e.g., Site Data + cleaned Surface Obs). Record provenance (lineage metadata + human note).
- Data integrity: ensure runs record which inputs they used (retain relationships between projects/objects and outputs) to support reproducibility.
- Data downloads: allowed on dataset-bearing objs: e.g., Grid Registration, Daymet Output; others too if they have files (DEM, masks, etc.). Elena: mark which ones are downloadable in the UI.
- Low-bandwidth viz only: quick-look visualizations for Surface Observation coverage, basic DEM hillshade, Analysis Mask outline, Site Data geometry previews. Priority = low. Jess: keep it minimal; no heavy GIS.

Daymet + Biome-BGC wiring
- Daymet must ingest surface observations, interpolate to gridded weather (over target projection/grid). Maya: “that’s the key step.”
- Support Daymet “project” concept: can define, run, and keep Daymet projects inside portal.
- Outputs from Daymet become inputs to Biome-BGC runs; keep the linkage visible in the portal.
- Compute: processing should run on remote nodes (Hemisphere cluster). Portal host remains lightweight.
- Globus: all transfers + remote job interactions integrated with the Globus toolkit; security via GSI; aim for robust behavior across the grid.
- NCAR storage: file-based storage lives in the Mass Storage System; scratch may be used for staging on compute side, but final outputs land in MSS. Priya: “Document this flow.”

Disturbances for BGC
- Need disturbance inputs used by Biome-BGC during simulation steps.
- Event types covered for v1: fires, deforestation.
- Disturbance project collects many events. Unlimited events — no hard coded max.
- Each event = an instantaneous point in time. Represent by date/time stamp. No durations. Jess: UI should not show start/end fields.
- Events included with the Biome-BGC input bundle; track which events went into a run.

User Management / Security / Roles
- Gatekeeper for login; portal prompts for credentials at session start; single sign-on style. No re-prompt on run submission or download (delegation behind the scenes).
- Only approved accounts can access. Admins manage approvals and roles.
- Scientists (favored): modeling features, create/run projects, create/edit domain objects (with rules above). Data Users (lowest priority): read-only to public/shared objects, can download permitted datasets, no run rights. Portal Admins: user lifecycle, unlock accounts, policy enforcement.
- Comply with NCAR security policies (password rules, access controls, secure transport, account lifecycle). Priya will send checklist.
