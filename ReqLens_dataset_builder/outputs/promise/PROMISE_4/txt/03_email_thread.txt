Quick answers + plumbing notes:

- History: yes, we’ll capture user + datetime for any create/modify, and we’ll add “system” entries when background jobs touch a case. Users can view the full history on the case details screen.

- Follow‑up: hiding actions that don’t fit the current status or the user’s role is the plan. View‑only users won’t see initiate/escalate.

- Letters: per Ravi, the Print Letter Utility schema is quirky but stable. We’ll validate before sending to avoid rejections.

- Duplicates: engineering will define the composite key (cardmember + merchant + transaction(s) [+ reason if required]) and block creation if there’s an existing open case that matches. The goal is to prevent duplicates entirely.

- Reason codes: we’ll wire to a service that computes the eligible set from transaction type + MCC so we avoid a global dump.

- Viewing existing cases: search results will deep link into case details with history and follow‑ups visible.

- Performance/availability tracked elsewhere: navigation around ~5 seconds; maintenance windows as agreed; concurrency target 350 concurrent sessions.

- Adjustments: when chargebacks trigger adjustments, we’ll route requests to Posting & Billing via API. Downstream posting may be asynchronous.

Open items:
- Need Compliance to finalize the transaction‑age gate for eligibility so we can enforce it in the UI.
- Nina — we may need to tweak color accents to meet contrast ratios while keeping the label/icon visible.
