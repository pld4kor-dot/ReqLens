Notes (messy):
- UI/UX:
  - Use native controls only on both platforms → no custom nav widgets; standard menus & buttons. Agents expect Back, Menu, soft keys to work. “This should look like a normal Palm app, not a game” (Leo).
  - Brand consistency: Brooke: keep our blue, gray, font sizes consistent across screens; professional look, no playful icons. Kira to align w/ brand guide. Fonts: map to CE/Palm stock fonts, keep consistent.
  - Zero training philosophy: terms familiar to Realtors: “CMA,” “comps,” “showing,” “listing,” “buyer,” not “entity”/“record.” Self-explanatory flow. “If they need a training video we blew it” (Brooke).
  - Ease of learning: New hire should pick up in minutes; minimal learning curve. Onboarding screens? [TBD — Brooke wants none; Kira suggests 2-screen tip]. NOT decided yet.
  - Install: Self-installable installer; no printed instruction sheet. Owen asked about QR code quick-start; Brooke says ok but no paper booklet. Goal: 90%+ of untrained agents succeed on first try. [Track metric].

- Search / Photos / Save:
  - MLS query from device — must support typical criteria (price range, beds, neighborhood), no weird query syntax. Use realtor-friendly labels.
  - Results need to feel fast. Priya wrote: “Search < 30 sec after submit” (metrics). Leo: “fast as humanly possible,” but ok with upper bound.
  - Show clear property images in results + detail. Avoid over-compression. Ensure tap-through to larger image view given device constraints.
  - Allow saving of search results; accessible later. Clarify: save criteria + result snapshot? [TBD]. Important for offline.
  - Area map view: show pins in specified area. At minimum: retrieve map showing property locations for chosen area.
  - Driving directions: tap a listing → route shown. Use built-in mapping or light directions.

- CMA:
  - Generate CMA in field. Quick comps selection → summary report. Timing: under ~60 (Dana said 60). “Acceptable time” for CMA = minute-ish, not coffee break. Owen: log the time from criteria entry to report.
  - CMA correctness: Use board data; respect permissions.
  - Cache generated CMAs for offline viewing later.

- Sync (office system):
  - Automatic sync cadence: every hour. “On the hour” job already exists on ops side. We will not change this. [Earlier idea of 10-min push — crossed-out].
  - Sync covers client contacts + appointments. After sync completes, device should notify user of new or changed appointments. (Example: “2 new showings added since 11:00”).
  - Perceived speed: sync should feel timely. Agents shouldn’t wonder “did it work?”. Show last synced time.
  - Downloaded items accessible offline later.
  - Confusion: Leo asked “can we pull immediately after someone hits save at office?” Priya said no → office system is hourly; the device can’t make it faster. [Open issue: whether a UI ‘Sync now’ button is acceptable — Brooke wary of UX clutter]. NOT decided yet.

- Offline + Availability + Prefs:
  - Offline mode: when connection drops, app still usable. View cached searches, previous CMA reports, and appointments.
  - Availability target: treat as 24/7. Priya wrote “999/1000 availability expectation” — roughly 99.9%. Brooke: “We show on Sundays. Don’t go down.”
  - Preferences: persist user settings frequently so crash/outage doesn’t wipe them. On restart, 100% of saved prefs restored exactly. QA to test crash scenarios.

- Security:
  - Access limited to registered, licensed agents only. No guest mode. Tie to brokerage directory + MLS where needed.
  - Every user authenticated AND authorized. Don’t conflate — AuthN vs AuthZ. Role checks for listing edit vs view.
  - Input validation everywhere to avoid bad data entering system (e.g., no negative prices, required fields for listings, date formats). Protect MLS and office systems from junk.

- Listings (create/update):
  - Agents want to update open house time, fix typos, change status. Create new listing from device also requested by Brooke (scope confirmed by Dana). Must respect MLS rules, picklists, required fields.
  - Edge cases: network drops during update → retry later; do not create duplicates. Owen to define.

- Scalability / Users:
  - Multi-remote-user support baseline. QA load testing: 1000 simultaneous agents expected today. Plan for growth to 5x (~5000) within a year. Priya: “Don’t melt the server during spring rush.”
  - Consider modest headroom for spikes. Logins at top of hour with sync — watch contended endpoints.

- Device constraints:
  - Target platforms: Windows CE and Palm OS only (for v1). Conform to native UI conventions. No custom gestures. Use platform-standard menus, buttons, dialog patterns.
  - Notifications: use platform-appropriate alerts; don’t simulate iOS banners.
  - Font/color: use consistent palette + font mapping. Designer to provide a mapping doc.

- Calls & Notifications:
  - In-app calling to buyers/sellers to coordinate showings.
  - Notify agent when client responds (call back, confirms/cancels). Works when the call starts from the app so we can detect it. Coordinate details with ops.

  - Brooke prefers zero onboarding steps; Kira suggests minimal tips. NOT decided yet.
  - Brooke wants “no lag at all” on images; Alex warns device memory limits. Agreement: prioritize clarity; may limit number of simultaneous thumbnails.
