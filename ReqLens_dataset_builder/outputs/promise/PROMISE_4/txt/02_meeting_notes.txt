Initiation — Ticket Retrieval Requests (TRR)
- Start a retrieval from an eligible transaction view (real‑time). Guided fields (wizard/stepper) collect required info; due date is computed from business rules (read‑only).
- TRR types: original receipt, copy, portfolio (portfolio = supplemental documents like a rental agreement). Tooltips to clarify. [Nina]
- Restriction: external users must NOT be able to request the “original” receipt type (internal‑only). Copy and portfolio allowed externally. [Policy]
- Confirmation after creation: display case number, TRR type (copy/original/portfolio), and merchant response due date.
- Letters: generate letter requests formatted to Letters app spec -> Print Letter Utility. [Ravi]
- Reason codes: context‑driven by transaction type + merchant MCC; do not show irrelevant codes.

Initiation — Chargeback
- Chargeback can be “pending” or “immediate.” Required prompts block submission until completed; collect reasons and document links as needed.
- Confirmation after submission: case number, subtype (pending vs immediate), and merchant response due date; also logged in the case timeline.
- Letters: send chargeback notification via Letters app -> Print Letter Utility (use exact schema/format).
- Adjustments: if an account credit/debit is needed, route the request to Cardmember Posting & Billing via API (no manual files).

Selecting transactions
- Only eligible transactions (age window per Reg/network). UI must block too‑old transactions. [Gloria to confirm exact day limit]
- Direct entry from transaction list or statement detail; prefill from Statement DB (no re‑keying).
- Multi‑transaction case: allow combining multiple transactions from the same merchant; enforce same‑merchant constraint.
- Duplicate prevention: the system must prevent duplicate case creation for an already‑open matching pattern (cardmember + merchant + transaction(s) [+ reason as needed]). Zero tolerance—block 100% of duplicates. [Define key signature]

Search & listing
- Search by: case number, merchant account number, cardmember account number, issuer number. Filters: date range, dispute type (TRR vs chargeback), case status (open/closed/all), reason code.
- Results: clearly distinguish type and status; provide color cues paired with labels/icons (accessibility). Pagination as needed.
- From results, open case details with history and follow‑ups.

Case history & follow‑up
- Maintain a full audit trail: each create/modify stamped with user + timestamp; system‑generated actions labeled “system.”
- Users can view the complete timeline/history on the case.
- Follow‑up actions menu filtered by case status and user access (e.g., hide actions that don’t apply to closed cases; view‑only users see fewer actions).
- System events (aging, auto updates) create activity entries and trigger an in‑app notification for the case creator upon next sign‑in. [Ben to confirm infra]

Integrations
- Source‑of‑truth: transaction details from Statement Database; cardmember details from Cardmember Information DB; merchant details from Merchant Information DB. Reference by IDs; no manual overrides.
- Letters app integration: submit TRR/chargeback requests in Letters app schema, handed to Print Letter Utility for printing. [Validate before submit]
- Posting & Billing: send adjustment requests via API; downstream posting may be asynchronous.

Security + Access (recap)

- Access model: View vs Update. Only Update role can initiate TRR/chargeback. External default = View.
