Notes (messy):
- Name: keep official product name as Qheadache. window title, About text, even Save file name if any (but we are NOT doing saves now) [TBD] —> but name Qheadache everywhere.
- Domain: sliding-block desktop puzzle. one “headache” puzzle only. UI should help you solve that particular layout (not multiple levels). People will try to solve faster/fewer moves.
- App type: standalone exe/app bundle. No server calls. local-only data. Single user per machine session (no profile switching in-app). [Renee asked: do we ever need user management?] Answer: No. Single local user only.
- UI approach: menus + graphics + optional audio. Must be fully playable with sound off or system muted. Keep it simple. No audio dependency.
- Main window: menu bar + board area + status region (time, actions). Keep menu-driven interactions. (No fancy context menu for now; keep top bar.)
- Board geometry: rectangle height 5x, width 4x. Black background. Fit inside window with some padding. x is layout unit. clamp x within 50–100 pixels. If window big, pad; if small, enforce min window to avoid x < 50.
- Colors: blocks yellow. Board black. (briefly debated inverted colors — crossed out) NOT decided yet (actually decided: keep black board, yellow blocks). [Update: stick to original plan]
- Spacing: blocks separated by ~0.1x “gutters”. So visually not touching. Keep consistent around all blocks. (Asked: margin to border as well? Possibly same 0.1x to outside? [TBD], but requirement is block-to-block spacing at least.)
- Preset set (fixed):
  * 4 small squares (1x by 1x)
  * 4 tall rectangles (2x high, x wide)
  * 1 wide rectangle (x high, 2x wide)
  * 1 big square (2x by 2x)
  counts are fixed. no variants. [Someone suggested more pieces — crossed out]
- Interaction model (pointer-based):
  * Left mouse down on a block => enter movement state (“block movement”).
  * While in movement, block should track cursor position. Constraint rules prevent overlap.
  * On left button release (on that selected block): return to selection state. (Note: confirm nothing odd on release off the board; still return to selection.)
  * People may use other pointing devices (trackpads etc.). Should still move the block with pointer; keyboard support for menus and entering name.
- Input devices: keyboard, mouse, touchpad, trackball, etc., anything that moves an on-screen cursor. Use keyboard for menu nav and for victory name entry. (No touch/gesture specifics in scope.)
- Resolution: 800x600 minimum. At that size, digits for time and action counts must be legible. Use Qt font scaling as needed. Renee to test on 800x600 Windows.
- Frameworks: Qt required. Cross-platform — runs on OS supported by Qt. Windows support is explicitly required. Also test on Linux; mac if possible.
- Undo/Redo: provide history up to last 1000 actions. Discussion: is 1000 enough? Yes. Not user-configurable now. Use ring buffer (Lars: implement). [Renee: ensure redo resets correctly when new moves are made.] 
- Progress tracking: show elapsed time during a game session. Show action count incrementing per move. Present near the board in a non-intrusive area. Keep digits large.
- On win (victory condition reached): prompt for player name. Record: player name + elapsed time + action count. Save locally. (No online sync.)
- Statistics: accessible via menu. “Statistics” menu contains two items. Order matters: Display first, Erase second. Accept both mouse and keyboard selection. On Display => open scores window with list of recorded players (all entries visible). On Erase => wipe stored stats (no confirmation for now; keep it simple). After performing either action, the Statistics menu should close — not remain open.
- Help menu: contains About item. Accept keyboard and mouse selection. On About => open About window. The Help menu closes after About is shown. About text must exactly read: Qheadache 1.0 by Jean-Philippe Brossat jp_brossat@yahoo.fr (Verify exact spaces/characters). [Renee to test literal text]
- Single-user: Only one user per machine context. Stats are shared on the machine (no per-user profiles). Note: QA to confirm behavior if OS user accounts differ; scope is single user in-app per session. [But we are not preventing multiple OS users; just no multi-profile in app.]
- Audience assumptions: No special qualifications required. Basic computer users only. Kids as young as 8 should manage without training. Adults (puzzle enthusiasts) also targeted.
- Menu behavior consistency: After executing an action (Stats->Display or Erase; Help->About), the menu should close. Avoid persistent dropdowns post-action. (Renee: add test case for that.)
