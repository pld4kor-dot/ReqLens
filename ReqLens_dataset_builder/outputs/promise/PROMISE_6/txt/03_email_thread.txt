All,

Security points we need to hold the line on:

1) Access control: Only employees or pre-approved external users can log in. No public sign-ups.
2) Manager-only search: Queries that pull reservations by user must be limited to manager roles.
3) Data integrity: Reservation data in the DB should reflect exactly what the user entered (no silent truncation/alteration).
4) Malicious inputs: Validate and sanitize. If someone sends garbage, do not crash and do not corrupt data.
5) Threat updates: Provide a way to deliver automatic updates as new threats emerge (process details can come later with Ops).
6) Logging: Capture exceptions and provide a way to transmit logs securely to vendor support for diagnosis.
7) Accessibility: Conform to disability access standards.

Also, please ensure the UI gives users clear confirmations when actions succeed. Silence creates tickets.

— Serena

Facilities items we don’t want to miss:

- Floorplans: Show a building map in the app with where the conference rooms are, plus basic equipment details on click.
- Easy map updates: We’re renovating. I don’t want to open a ticket to add new maps—let us drop in new ones or configure them with minimal help.
- Equipment transport: When mobile equipment is reserved, create a transport request and notify building personnel to move it. Track statuses (requested, in transit, delivered, returned).
- Availability: People ask “Is anything open now?”—show an “Available” status for rooms and let them filter for what’s free.
- Rooms/equipment mgmt: We add/update/delete rooms and gear constantly. Make it straightforward. And when equipment is reserved, associate it with the meeting.

Gripe: Our old tool updated the room calendar hours later. Here, as soon as someone books, the schedule needs to reflect it so we avoid conflicts.

Miguel et al.,

Great list. Here’s how the design aligns with that and our broader plan:

- Meeting model: We’ll store entries with an agenda (and keep updates), a meeting type (for reporting), and attach organizer contact info.
- Invitations: Organizers can invite other employees. Notifications go out via email at minimum; the framework lets users indicate channel preferences as we expand.
- Acknowledgements: We’ll record attendee responses.
- Schedule consistency: When a room is reserved, the schedule will reflect the reservation immediately and avoid conflicts.
- Search results: Filter rooms and equipment by parameters, and display matches—including a clear “Available” indicator for rooms that are free.
- Transport: We’ll record equipment transport requests, notify the facilities distribution list, and track status so you can see progress.
- Maps: We’ll display the building floorplan with room locations. Plan is a configuration-driven approach so you can bring new maps online during the product lifecycle with minimal assistance.

On tech stack:

- DBMS: The product supports Oracle, SQL Server, DB2, MySQL, HSQL, and MS Access. The DB can live on the same machine as the app or on a separate host. The DB layer is configurable—no code changes to swap engines.
- App server: We’ll deploy on Tomcat primarily; JBoss also works. Java SE/EE for the implementation.
- Browser/Email: HTML UI tested across a broad set of browsers; we’ll aim for solid rendering across the majority. The app will use corporate email servers to send messages.

We’re also building in exception logging with a way to transmit logs when needed, delivering automatic security updates, and meeting accessibility standards. Performance/uptime targets are being tracked with Ops.

Regards,

Alex
