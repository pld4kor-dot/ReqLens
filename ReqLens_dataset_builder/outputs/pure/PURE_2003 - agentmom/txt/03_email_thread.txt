Hi all,

Quick recap after standup:

- Compatibility stays on by default so new agents can interoperate with agentMom 1.2. Your legacy nodes should still send/receive basic messages without code changes. For features that require shared understanding (e.g., per-message protection), use them where both sides support them; otherwise keep those particular flows plain.
- Security: per-message protection remains optional. You can flip it on for unicast; the receiver will handle protected messages automatically when configured. Same idea for multicast groups that opt into protection.
- Group membership: after leave(), deliveries from that group stop without a grace period. For a temporary pause, use opt-out; deliveries halt while paused and resume when re-enabled.
- Broadcast: behavior is scoped to the local network. If you broadcast a ping, all hosts on that LAN should see it, but it won’t go beyond. It’s intended for discovery/announcements rather than sensitive content.
- Substitution: we plan to show a replacement agent joining a group and taking over the role so the group can continue without manual reconfiguration.
- Demo: We’ll keep the “starred” list tight per Dr. Chen. I’ll circulate the list once we finalize it in the SRS draft.

If anything here conflicts with your expectations, please shout.

–Maya

QA plan snapshot so it’s written down:

- Unicast protection toggle: when sender flags “protect=true,” the receiver (with config) should hand plaintext to the app; when false, plaintext end-to-end. We’ll verify no spurious failures when protection is off.

- Multicast protection: when a group is configured for it, members (with config) should handle protected messages on receipt. If a node isn’t in the group, our app layer should not treat that traffic as consumable.

- Leave() semantics: deliveries stop immediately after leave(). I’ll assert no group payloads arrive after the leave call.

- Broadcast scope: set up two subnets to confirm broadcast does not cross. On one subnet, every host will receive the ping; on the other, none should.

- Mixed-version tests: 1) 1.2 -> new unicast (plain), 2) new -> 1.2 unicast (plain), 3) new <-> new unicast protected and plain, 4) group membership enforcement around join/leave across versions.

- Substitution: simulate failure of an agent in a group, then start a replacement; verify the group continues and that members see the join/leave notifications around the swap.

If I missed something, reply-all.

— Sam
