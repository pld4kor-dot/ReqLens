Hi all,

On message signing/encryption: we can support WS-Security headers, but be aware it adds processing overhead. If you’re also holding to the ~2s end-to-end target, we’ll need multi-threading and load balancing across the hub’s workers so the crypto doesn’t become the bottleneck.

We’ve also seen trouble when hubs try to store raw payloads for a year without planning. If you’re retaining, please make sure access to archives is controlled and aligns with city policy.

Also, I want to raise a design point: failure handling. If a work order create call succeeds in EAM but the hub misses the ack due to a network blip, will you support a compensating mechanism? The goal would be to confirm final state and avoid duplicates while closing the loop. We can coordinate on what our side exposes to help you validate state.

Regarding upgrades: we can align with rolling updates. Hot-adding workers is fine; just register them with the LB. Totally okay from our perspective.

- Carlos

Team,

Thanks for the inputs. Capturing where we landed (and a couple of open points):

Security
- Authentication + authorization: portal users are role-based; service accounts scoped with least privilege. No anonymous access.
- Message privacy + integrity: adopt a standards-based approach (WS-Security or equivalent) so payloads are encrypted/signed. We’ll include integrity checks to catch tampering, not rely only on TLS.

Audit/Retention
- Retain message history + metadata for minimum 1 year. Include correlation IDs, timestamps, actions/retries, and any transformations.

Failure handling
- Use retry/backoff and support compensating transactions where a step succeeded on one side but not acknowledged. Coordinate with EAM behavior to validate end-state and avoid duplicates.
- Route around failing endpoints when possible; do health checks and failover to healthy nodes (on the EAM side where supported).

Platform/Extensibility
- Platform independent design: no lock-in to a single OS or dev environment. Deployable on Windows/Linux/containers as needed. Adapters/connectors should be cross-platform.
- Accept new technologies later without core architecture changes (plugin/adapter model).

Operations
- Adding new processing capacity (more workers) must not interrupt service. We will do hot-add with LB registration.
- We continue to target around 2s end-to-end handling and frequent (about once per second) intake checks; concurrency + LB are assumed to keep us there even with security overhead.

Open items / pushback
- Carlos flagged crypto overhead; we’ll mitigate via concurrency but will baseline perf in perf testing. If we need to tune, we’ll do so without relaxing security requirements.
- Janet to schedule a security workshop to nail down the WS-Sec profile and key management approach.
- Priya to propose the archive schema and access controls for audit queries.
- Maya to verify with Wendy that our UI and API choices still align with the city framework (no off-framework widgets; this is a separate topic but interrelated).

Partial resolution: we’re aligned on the need for role-based access, message-level security, one-year retention, compensating + routing around failures, and platform independence. Remaining action items above. I’ll incorporate this into the requirements draft and circulate by Friday.

- Maya
